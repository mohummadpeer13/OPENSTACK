Démystifier OpenStack

Avant l’arrivée d’OpenStack, gérer une infrastructure informatique signifiait intervenir directement sur les serveurs physiques : installer les systèmes, configurer les disques, brancher les réseaux… Chaque opération demandait du temps et des manipulations manuelles.

OpenStack, c’est la couche logicielle qui prend le contrôle de tout ce matériel pour le transformer en un Cloud privé.

Face à ces contraintes, des solutions comme OpenStack ont été développées pour automatiser et simplifier la gestion des ressources. Il permet de transformer une infrastructure physique en un environnement cloud, où les machines peuvent être créées, modifiées et supprimées en quelques clics, sans intervention directe sur le matériel.

Grâce à lui, vos processeurs, votre RAM et vos disques durs deviennent des ressources virtuelles que vous pouvez piloter, découper et distribuer à vos utilisateurs en quelques secondes depuis une interface web.

De la Virtualisation au Cloud

La Virtualisation est la fondation : elle permet de découper un serveur physique en plusieurs "tranches" (VM). Mais gérer 500 serveurs à la main reste un cauchemar.

Le Cloud Computing, c'est l'étape supérieure. C'est l'automatisation totale de ces ressources. On ne parle plus de gérer des machines, mais de consommer des services via une API ou une interface, sans jamais toucher au matériel.

Le Cloud Operating System

"Comme Windows gère votre PC, OpenStack gère des datacenters entiers. RAM, CPU et Disques de milliers de serveurs fusionnent pour devenir un seul ordinateur géant."

C’est quoi Nova ?

Compute as a Service

Nova est le service de calcul d’OpenStack. Il permet de créer, gérer et supprimer des machines virtuelles (instances).

💡 Quel est son rôle ?

Nova est responsable du cycle de vie des machines virtuelles.

👉 Il décide où lancer la VM, démarre la machine et gère son état

👉 Rôle principal : exécuter et gérer les machines virtuelles

Nova ne travaille pas seul : il utilise d’autres services comme Glance (images), Neutron (réseau) et Cinder (stockage).

💡 Exemple concret :

📦 Tu choisis une image (Glance)

⚙️ Tu lances une VM avec Nova

🌐 Neutron lui donne un réseau

💾 Cinder peut ajouter un disque

🚀 Résultat : une machine virtuelle prête à être utilisée

⚙️ Instances
création et gestion des VM
🚀 Déploiement
lancement rapide de machines
🔄 Cycle de vie
start, stop, reboot, delete
🧠 Analogie :
Nova est comme un chef d’orchestre :
  • • Il coordonne les ressources
  • • Il lance les machines
  • • Il travaille avec les autres services

👉 Nova est le cœur du cloud

Service de calcul responsable des machines virtuelles OpenStack

Ce que permet Nova :

  • • Créer, démarrer, arrêter, redimensionner ou supprimer des machines virtuelles en libre-service
  • • Migrer des VMs d’un serveur à un autre sans interruption
  • • Gérer des serveurs physiques (bare metal) via Ironic
  • • Supporter de nombreux hyperviseurs : KVM, Xen, VMware, Hyper-V, etc.

Comment Nova est-il construit ?

Nova est conçu pour être très scalable et fiable. Il repose sur une architecture basée sur des messages.

Tous les composants de Nova communiquent entre eux via une file d’attente de messages (Message Queue).

Message Queue
Généralement RabbitMQ ou Redis

Cette architecture apporte plusieurs avantages clés :

  • Scalabilité horizontale
    On peut ajouter facilement de nouveaux serveurs sans tout casser.
  • Tolérance aux pannes
    Si un composant tombe, les autres continuent de fonctionner.
  • Livraison garantie des messages
    Aucun message n’est perdu, même en cas de surcharge.
  • Traitement par priorité
    Les tâches urgentes sont traitées en priorité.

Grâce à ce système de file d’attente, Nova peut gérer des milliers de machines virtuelles sur des dizaines de serveurs physiques sans problème de performance.

Les principaux composants internes de Nova

Nova n’est pas un seul gros bloc monolithique. Il est composé de plusieurs services spécialisés qui travaillent ensemble de manière coordonnée.

Nova API

C’est le point d’entrée principal de Nova. Il reçoit toutes les demandes via HTTP (création de VM, arrêt, redimensionnement…) venant des utilisateurs, du Dashboard Horizon ou d’autres services OpenStack.

Nova Scheduler

L’ordonnanceur intelligent de Nova. Il analyse les ressources disponibles sur tous les serveurs physiques et décide sur quel nœud il est le plus judicieux de placer chaque nouvelle machine virtuelle (en fonction des règles, filtres et disponibilités).

Nova Conductor

Il joue le rôle d’intermédiaire sécurisé entre les autres composants et la base de données. Il empêche les serveurs de calcul d’accéder directement à la BDD, ce qui améliore la sécurité et les performances.

Nova Compute

Installé sur chaque nœud de calcul (serveur physique). C’est lui qui parle directement à l’hyperviseur (KVM, Xen, VMware…) pour créer, démarrer, arrêter et gérer concrètement les machines virtuelles.

Base de données SQL

Stocke l’état complet du système : toutes les instances en cours, les flavors, les métadonnées, l’historique, etc. Tous les composants passent par Nova Conductor pour y accéder.

Ces composants communiquent entre eux via une file d’attente de messages (RabbitMQ), ce qui rend Nova très scalable et résilient.

Avec qui Nova interagit-il ?

Nova ne travaille jamais seul. Il communique en permanence avec plusieurs services clés d’OpenStack pour fournir une expérience complète et cohérente.

Keystone

Authentification & Gestion des accès
Avant toute action, Nova interroge Keystone pour vérifier l’identité de l’utilisateur (token). Keystone détermine ensuite les droits associés (projets, rôles) afin d’autoriser ou refuser les opérations comme la création, la modification ou la suppression d’une machine virtuelle.

Glance

Catalogue d’images système
Nova utilise Glance pour récupérer les images des systèmes d’exploitation (Ubuntu, CentOS, Windows…). Lors du lancement d’une instance, l’image est téléchargée puis utilisée comme base pour créer le disque de la machine virtuelle.

Neutron

Réseaux virtuels & connectivité
Nova s’appuie sur Neutron pour connecter les machines virtuelles aux réseaux. Neutron crée les interfaces réseau (ports), attribue des adresses IP et applique les règles de sécurité (groupes de sécurité, routage, DHCP).

Cinder

Stockage bloc persistant
Nova peut attacher des volumes fournis par Cinder aux machines virtuelles. Ces disques persistants permettent de conserver les données indépendamment du cycle de vie de l’instance (redémarrage, suppression, migration).

Placement

Gestion intelligente des ressources
Placement fournit à Nova une vision précise des ressources disponibles (CPU, RAM, disque, GPU) sur l’ensemble de l’infrastructure. Il permet de choisir le serveur le plus adapté pour lancer une instance en fonction des contraintes et de la capacité réelle.

Toutes ces interactions reposent sur des appels API REST et une file de messages (RabbitMQ). Nova agit comme un orchestrateur central qui coordonne les services pour transformer des ressources physiques en machines virtuelles utilisables à la demande.

C’est quoi Neutron ?

Networking as a Service

Neutron est le service de réseau d’OpenStack. Il permet de créer et gérer toute l’infrastructure réseau du cloud : réseaux, sous-réseaux, routeurs et adresses IP.

💡 À quoi sert Neutron ?

Neutron permet de connecter les machines virtuelles entre elles et avec l’extérieur (Internet).

👉 Il agit comme un réseau virtuel programmable

👉 Rôle principal : connecter les ressources et gérer la communication réseau

Sans Neutron, les machines virtuelles ne pourraient pas communiquer entre elles ni accéder à Internet.

💡 Exemple concret :

🖥️ Tu crées 2 machines virtuelles

🌐 Tu les connectes à un même réseau

🔁 Elles peuvent communiquer entre elles

🌍 Avec une IP publique → accès Internet

🌐 Réseaux
création de réseaux isolés
🔀 Routage
communication entre réseaux
🌍 Accès Internet
gestion des IP publiques
🌐 Analogie :
Neutron est comme un réseau d’entreprise :
  • • Les VM = ordinateurs
  • • Les réseaux = LAN
  • • Les routeurs = passerelles
  • • Les IP publiques = accès Internet

👉 Neutron est donc le cœur réseau du cloud

Gestion complète du réseau et des communications dans OpenStack

Ce que permet Neutron :

  • • Créer des réseaux privés (tenant networks) et des sous-réseaux
  • • Attribuer des adresses IP aux machines virtuelles
  • • Créer des routeurs virtuels et connecter plusieurs réseaux
  • • Fournir des Floating IPs pour accéder aux VMs depuis Internet
  • • Configurer des Security Groups (firewalls)
  • • Supporter des technologies avancées : VLAN, VXLAN, GRE, etc.

Comment Neutron est-il construit ?

Neutron est un service standalone et hautement scalable. Il déploie plusieurs processus sur différents nœuds (controller, network nodes et compute nodes).

Tous ces processus communiquent entre eux via une file d’attente de messages (RabbitMQ) et avec la base de données. Neutron est très modulaire grâce à son système de plugins et mechanism drivers (ML2 avec Open vSwitch, OVN, etc.).

Les principaux composants de Neutron

Neutron Server

Le cœur du service. C’est un démon Python qui expose l’API REST. Il reçoit toutes les demandes réseau et communique avec la base de données.

L2 Agent (ex: OVS Agent)

Tourne sur chaque nœud de calcul. Il configure le commutateur virtuel local (Open vSwitch, Linux Bridge…) pour connecter les machines virtuelles au réseau.

DHCP Agent

Fournit le service DHCP aux réseaux. Il attribue automatiquement des adresses IP aux machines virtuelles.

L3 Agent

Gère le routage (Layer 3) : routeurs virtuels, NAT, Floating IPs et l’accès vers les réseaux externes.

Note : Avec le driver OVN (moderne), beaucoup d’agents traditionnels sont remplacés par ovn-controller, ce qui simplifie l’architecture.

Avec qui Neutron interagit-il ?

Neutron est le service réseau d’OpenStack. Il ne crée pas de machines, mais fournit toute l’infrastructure réseau nécessaire pour connecter les ressources entre elles (VM, stockage, internet…).

Nova

Neutron fournit à Nova les interfaces réseau (ports) pour connecter les machines virtuelles (IP, DHCP, sécurité, etc.).

Keystone

Neutron s’appuie sur Keystone pour authentifier les utilisateurs et contrôler l’accès aux réseaux, sous-réseaux et routeurs.

Glance

Indirectement via Nova : les images de VM lancées utilisent automatiquement les réseaux configurés par Neutron.

Cinder

Indirectement via Nova : les volumes de stockage sont attachés à des machines virtuelles déjà connectées au réseau Neutron.

Neutron agit comme un architecte réseau : il crée et gère les réseaux, et permet à tous les services (Nova, Cinder, etc.) de communiquer entre eux de manière sécurisée et flexible.

C’est quoi Glance ?

Image Service

Glance est le service de gestion des images d’OpenStack. Il permet de stocker, gérer et distribuer des images de machines virtuelles.

💡 C’est quoi une image ?

Une image est un modèle prêt à l’emploi contenant un système d’exploitation (Linux, Windows) et parfois des logiciels déjà installés.

👉 C’est à partir de cette image que les machines virtuelles sont créées.

👉 Rôle principal : fournir des images à Nova pour créer des machines virtuelles

Quand tu lances une VM, Nova va chercher une image dans Glance pour démarrer la machine avec un système déjà prêt.

💡 Exemple concret :

📦 Tu upload une image Ubuntu dans Glance

⚙️ Tu crées une VM avec Nova

🚀 La VM démarre avec Ubuntu déjà installé

📦 Images VM
systèmes prêts à être lancés
🚀 Déploiement rapide
création instantanée de VM
🔁 Réutilisable
même image pour plusieurs VM
📀 Analogie :
Glance est comme une bibliothèque d’images ISO :
  • • Tu stockes tes systèmes (Linux, Windows…)
  • • Tu choisis une image
  • • Tu démarres une machine avec

👉 Glance est donc la source de démarrage des machines virtuelles

Service de gestion et distribution d’images pour les VM OpenStack

Ce que permet Glance :

  • • Stocker et gérer les images de machines virtuelles
  • • Fournir des images prêtes à l’emploi (Ubuntu, Debian, Rocky Linux, Windows…)
  • • Gérer les snapshots des machines virtuelles
  • • Permettre aux utilisateurs de créer et d’uploader leurs propres images personnalisées
  • • Stocker les images à distance (Swift, Ceph, S3…) pour économiser l’espace disque local

Comment Glance est-il construit ?

Glance est un service relativement léger et bien structuré. Il sépare clairement la gestion des images de leur stockage physique, ce qui le rend flexible et scalable.

Glance API

C’est le point d’entrée de Glance. Il expose une API REST qui permet à Nova, Horizon et aux utilisateurs de lister, uploader, télécharger, supprimer ou rechercher des images. Il gère également l’authentification via Keystone.

Glance Store

C’est la partie qui s’occupe du stockage réel des images. Il agit comme une couche d’abstraction entre Glance et les différents systèmes de stockage.

Il supporte plusieurs backends de stockage :

• Swift (le plus recommandé)
• Ceph
• Amazon S3
• Système de fichiers local

Grâce à cette architecture modulaire, Glance peut facilement s’adapter à différents environnements de stockage tout en restant simple d’utilisation pour les autres services comme Nova.

Avec qui Glance interagit-il ?

Nova

Consommateur d’images
Nova interroge Glance pour récupérer les images de systèmes d’exploitation. Lors du lancement d’une instance, l’image sélectionnée est téléchargée puis utilisée comme base pour créer le disque de la machine virtuelle.

Keystone

Authentification & contrôle d’accès
Glance s’appuie sur Keystone pour valider les requêtes via des tokens d’authentification. Il gère également les permissions afin de contrôler qui peut voir, télécharger ou modifier les images selon les projets et rôles.

Swift / Ceph / S3

Stockage des images
Glance ne stocke pas directement les images : il délègue leur stockage à des services externes comme Swift, Ceph ou S3. Cette séparation permet une meilleure scalabilité, une haute disponibilité et une gestion efficace des volumes de données.

C’est quoi Cinder ?

Block Storage as a Service

Cinder est le service de stockage bloc d’OpenStack. Il permet de créer des volumes (disques virtuels) que l’on peut attacher à des machines virtuelles.

💡 C’est quoi le stockage bloc ?

Imagine que tu branches un disque dur externe sur ton ordinateur. Cinder fait exactement la même chose dans le cloud : il fournit des volumes persistants que tu peux attacher à une machine virtuelle.

Contrairement au stockage objet (Swift), le stockage bloc est conçu pour être utilisé comme un disque classique :

  • • Formatage (ext4, NTFS…)
  • • Installation d’un système de fichiers
  • • Utilisation pour bases de données
  • • Applications critiques et données transactionnelles
👉 Rôle principal : fournir des disques persistants aux machines virtuelles

Les volumes Cinder sont persistants : les données restent même si la machine virtuelle est arrêtée ou supprimée.

💡 Exemple concret :

👉 Une VM héberge une base de données

💾 Les données sont stockées sur un volume Cinder

🔁 Si la VM est supprimée → les données restent intactes

💾 Stockage bloc
fonctionne comme un disque dur classique
🔗 Attachable
connectable à une ou plusieurs VM
🔒 Persistant
données conservées indépendamment de la VM
💽 Analogie :
Cinder est comme un disque dur externe dans le cloud :
  • • Tu le branches à une machine
  • • Tu stockes tes données dessus
  • • Tu peux le débrancher et le réutiliser ailleurs

👉 Idéal pour les bases de données et applications critiques

Stockage bloc persistant pour les machines virtuelles OpenStack

Ce que permet Cinder :

  • • Créer des volumes (disques virtuels) de n’importe quelle taille
  • • Attacher et détacher ces volumes à des machines virtuelles (comme un disque dur externe)
  • • Créer des snapshots de volumes pour faire des sauvegardes
  • • Étendre (agrandir) un volume existant
  • • Réutiliser un volume sur une autre instance après la suppression de la précédente
  • • Utiliser différents types de stockage (SSD, HDD, Ceph, etc.)

Comment Cinder est-il construit ?

Cinder est un service modulaire qui sépare clairement la gestion des volumes de leur stockage réel. Cette architecture lui permet de supporter de nombreux types de stockage en même temps (Ceph, LVM, SSD, stockage entreprise, etc.).

1. Cinder API

Point d’entrée des requêtes
La Cinder API expose une interface REST utilisée par Horizon, Nova ou les utilisateurs pour gérer les volumes. Elle reçoit les demandes (création, attachement, détachement, suppression) puis les transmet aux composants internes de Cinder pour exécution.

2. Cinder Scheduler

Choix du backend de stockage
Le Cinder Scheduler analyse les ressources disponibles (capacité, type de stockage, performances) sur les différents backends. Il sélectionne ensuite le backend le plus adapté pour créer le volume en fonction des besoins et des contraintes définies.

3. Cinder Volume Service

Exécution des opérations sur les volumes
Le Cinder Volume Service réalise les actions concrètes sur les volumes. Il communique avec les backends de stockage (Ceph, LVM, NetApp, etc.) pour créer, attacher, détacher et supprimer les volumes, tout en assurant leur gestion et leur disponibilité.

Grâce à cette architecture modulaire, Cinder peut travailler avec plusieurs technologies de stockage en même temps et s’adapter facilement à l’évolution de ton infrastructure.

Avec qui Cinder interagit-il ?

Nova

Gestion des volumes
Nova communique avec Cinder pour attacher ou détacher des volumes de stockage aux machines virtuelles. Cette opération permet d’ajouter un disque persistant à une instance, sans dépendre de son disque système, facilitant ainsi la gestion des données et leur conservation.

Keystone

Authentification & contrôle d’accès
Cinder s’appuie sur Keystone pour vérifier l’identité des utilisateurs via des tokens et contrôler les permissions. Il détermine ainsi qui peut créer, attacher, modifier ou supprimer des volumes de stockage selon les projets et les rôles.

Glance

Source d’images pour les volumes
Cinder peut utiliser des images stockées dans Glance pour créer des volumes bootables. Cela permet de provisionner rapidement un disque contenant déjà un système d’exploitation ou des données préconfigurées.

C’est quoi Swift ?

Service de stockage objet

Swift est le service de stockage objet d’OpenStack. Il permet de stocker et récupérer des fichiers (images, vidéos, sauvegardes…) de manière distribuée et scalable.

👉 Son rôle : stocker des fichiers de manière illimitée et distribuée

Contrairement à un disque classique (Cinder), Swift ne fonctionne pas avec des volumes attachés à une machine, mais avec des objets accessibles via API.

💡 Exemple : vous uploadez une image ou une vidéo

👉 Swift la stocke dans le cloud et vous fournit une URL pour y accéder

📂 Pas besoin de serveur → accès direct via API ou HTTP

📦

Stockage objet : fichiers stockés comme objets indépendants

🌍

Distribué : données réparties sur plusieurs serveurs

♾️

Scalable : capacité quasi illimitée

📦 Analogie :
Swift fonctionne comme un Google Drive ou Amazon S3 :
  • • Vous déposez des fichiers
  • • Ils sont stockés dans le cloud
  • • Vous y accédez via un lien

👉 Swift est donc un stockage en ligne massif, indépendant des machines virtuelles.

Stockage objet distribué et scalable pour OpenStack

Ce que permet Swift :

  • • Stocker des quantités massives de données non structurées (images, vidéos, logs, backups, documents…)
  • • Accéder aux fichiers de n’importe où dans le monde via une API simple
  • • Garantir une très haute disponibilité et tolérance aux pannes
  • • Faire scaler le stockage presque à l’infini sans interruption
  • • Vérifier l’intégrité des données avec des checksums automatiques

Comment Swift est-il construit ?

Swift est conçu dès le départ pour être extrêmement scalable, redondant et tolérant aux pannes. Il distribue les données sur de nombreux serveurs physiques.

Proxy Server

Point d’entrée du stockage
Le Proxy Server reçoit toutes les requêtes des clients et les distribue vers les nœuds de stockage appropriés. Il agit comme un intermédiaire central, assurant le routage, la sécurité et l’équilibrage des requêtes.

Account Server

Gestion des comptes
L’Account Server gère les comptes utilisateurs et maintient la liste des containers associés à chaque compte. Il permet de savoir quels containers appartiennent à quel utilisateur.

Container + Object Servers

Stockage et organisation des données
Les Container Servers gèrent l’organisation des objets (liste, métadonnées), tandis que les Object Servers stockent réellement les données. Ils assurent également la réplication automatique pour garantir la disponibilité et la durabilité des fichiers.

Swift utilise un algorithme de répartition appelé Ring qui permet de savoir très rapidement sur quels serveurs se trouve chaque objet.

Avec qui Swift interagit-il ?

Glance

Stockage des images
Glance utilise souvent Swift comme backend de stockage pour enregistrer les images de machines virtuelles. Cette intégration permet de bénéficier d’un stockage distribué, scalable et hautement disponible.

Cinder

Stockage des sauvegardes
Cinder peut utiliser Swift comme backend pour stocker les sauvegardes de volumes. Cela permet de conserver des copies sécurisées des données dans un stockage distribué, indépendant des volumes actifs.

Keystone

Authentification & contrôle d’accès
Swift s’appuie sur Keystone pour authentifier les utilisateurs via des tokens et gérer les permissions. Cela permet de contrôler l’accès aux conteneurs et aux objets selon les projets et les rôles définis.

C’est quoi Keystone ?

Service d’identité et de sécurité

Keystone est le service d’ authentification et d’autorisation d’OpenStack. Il gère les utilisateurs, les mots de passe, les rôles et les accès aux ressources du cloud.

👉 Son rôle : contrôler qui peut faire quoi dans le cloud

Chaque fois qu’un utilisateur ou un service veut effectuer une action (créer une VM, accéder à un réseau…), Keystone vérifie son identité et ses permissions avant d’autoriser l’accès.

💡 Exemple : vous cliquez sur “Créer une VM”

👉 Keystone vérifie : “Cet utilisateur a-t-il le droit de créer une machine ?”

✔️ Si oui → la requête est envoyée à Nova ❌ Sinon → accès refusé

🔐

Authentification : vérifie l’identité (login / mot de passe, token…)

🛡️

Autorisation : contrôle les permissions via les rôles (RBAC)

👥

Gestion des utilisateurs : projets (tenants), groupes et quotas

🏢 Analogie :
Si OpenStack est un immeuble sécurisé :
  • • Nova = les bureaux (machines)
  • • Neutron = les couloirs (réseau)
  • • Cinder = les archives (stockage)

👉 Keystone est le gardien à l’entrée : il vérifie votre badge et décide où vous pouvez aller.

Gestion des identités et contrôle des accès OpenStack

Ce que permet Keystone :

  • • Gérer les utilisateurs, groupes et mots de passe
  • • Créer et organiser les projets (tenants)
  • • Définir les rôles et permissions (RBAC)
  • • Générer des tokens d’accès sécurisés
  • • Fournir le catalogue des services (Service Catalog)
  • • Gérer plusieurs domaines (multi-domain)

Comment Keystone est-il construit ?

Keystone est relativement léger et très performant. Il est basé sur une architecture modulaire.

Keystone API

Point d’entrée de l’identité
La Keystone API (Identity v3) est utilisée par tous les services et utilisateurs pour s’authentifier. Elle valide les identifiants et délivre un token d’accès, utilisé ensuite pour autoriser les requêtes vers les autres services OpenStack.

Backend de Stockage

Stockage des identités
Keystone s’appuie sur un backend pour stocker les utilisateurs, projets, rôles et permissions. Il peut utiliser une base de données SQL (MariaDB/MySQL) ou un annuaire externe comme LDAP / Active Directory pour centraliser la gestion des identités.

Token Provider

Gestion des tokens d’accès
Le Token Provider génère et valide les tokens utilisés pour authentifier les requêtes. Il supporte différents formats (UUID, Fernet), avec Fernet recommandé car sécurisé, léger et ne nécessitant pas de stockage en base de données.

Comment Keystone interagit avec les services ?

1. Authentification
L’utilisateur envoie ses identifiants à Keystone. Si ils sont valides, Keystone génère un token d’accès.
2. Distribution du token
Ce token agit comme un badge de sécurité temporaire permettant d’accéder aux services.
3. Appel des services
L’utilisateur utilise ce token pour faire des requêtes vers Nova, Neutron, Cinder, etc.
4. Vérification
Chaque service contacte Keystone pour vérifier :
  • la validité du token
  • les permissions de l’utilisateur
5. Autorisation
Si l’utilisateur a les droits → action autorisée ✅ Sinon → accès refusé ❌

Keystone agit comme un agent de sécurité : il vérifie ton identité, te donne un badge (token) et tous les services vérifient ce badge avant d’exécuter une action.

Heat est le service d’ orchestration d’OpenStack. Il permet d’automatiser le déploiement complet d’une infrastructure cloud (machines, réseaux, stockage…) en une seule opération.

👉 Son rôle : transformer une infrastructure complexe en déploiement automatique

Au lieu de créer manuellement chaque ressource (VM, réseau, volume…), vous décrivez tout dans un fichier template (YAML/JSON).

💡 Exemple : vous voulez une application complète ?

👉 1 VM + 1 réseau + 1 volume + 1 IP publique

Avec Heat → un seul fichier → un seul clic → tout est créé automatiquement

⚡

Automatisation : déploiement complet en une seule commande

📄

Templates : infrastructure décrite en code (Infrastructure as Code)

🔁

Reproductibilité : déployer plusieurs fois la même architecture facilement

🏗️ Analogie :
Si OpenStack est un chantier :
  • • Nova = ouvriers (machines)
  • • Neutron = routes (réseau)
  • • Cinder = matériaux (stockage)

👉 Heat est le chef de chantier : il coordonne tout et construit automatiquement selon un plan.

Automatisation complète de l’infrastructure OpenStack

Ce que permet Heat :

  • • Déployer des infrastructures complexes en un seul clic (plusieurs VMs + réseaux + volumes + load balancers…)
  • • Définir toute l’infrastructure avec des templates (fichiers YAML)
  • • Mettre à jour ou supprimer toute une stack en une seule opération
  • • Créer des stacks imbriquées (nested stacks) pour réutiliser des modèles
  • • Mettre en place de l’auto-scaling automatique selon la charge
  • • Gérer le cycle de vie complet d’une application (Infrastructure as Code)

Comment Heat est-il construit ?

Heat est composé de plusieurs services qui travaillent ensemble pour transformer tes templates en ressources réelles.

1. Heat API

Point d’entrée de l’orchestration
La Heat API reçoit les templates (HOT) et les commandes (create, update, delete stack) depuis Horizon, l’API ou la CLI. Elle valide les requêtes puis les transmet au moteur Heat pour orchestrer le déploiement des ressources.

2. Heat Engine

Moteur d’orchestration
Le Heat Engine interprète le template (HOT), planifie les ressources et exécute les actions nécessaires. Il appelle les services OpenStack (Nova, Neutron, Cinder, Glance…), gère les dépendances entre ressources et suit en continu l’état de la stack (création, mise à jour, suppression).

3. Templates (HOT)

Infrastructure as Code (IaC)
Les templates HOT (Heat Orchestration Template) sont des fichiers YAML qui décrivent l’infrastructure complète (VM, réseaux, volumes, etc.). Ils définissent les ressources, leurs propriétés et leurs dépendances, permettant à Heat de déployer et gérer automatiquement toute la stack.

Heat transforme une simple description en une infrastructure réelle et cohérente.

Avec qui Heat interagit-il ?

Heat est le service d’orchestration d’OpenStack. Il ne crée rien directement : il coordonne les autres services en exécutant des templates (HOT) pour déployer automatiquement une infrastructure complète.

Nova

Gestion du cycle de vie des VM
Heat pilote Nova pour créer, démarrer, redimensionner et supprimer des machines virtuelles en fonction du template. Il applique les paramètres définis (image, flavor, réseau, nombre d’instances) et suit leur état tout au long du déploiement de la stack.

Neutron

Orchestration réseau
Heat pilote Neutron pour créer et configurer toute l’infrastructure réseau : réseaux, sous-réseaux, routeurs, ports et IP flottantes. Il gère aussi les connexions entre ressources (VM ↔ réseau) afin d’assurer la connectivité complète de la stack.

Cinder

Stockage bloc persistant
Heat pilote Cinder pour créer, attacher et gérer des volumes de stockage persistants. Les volumes sont automatiquement associés aux machines virtuelles et conservent les données indépendamment du cycle de vie des VM.

Glance

Catalogue d’images
Heat utilise Glance pour récupérer les images système (Linux, Windows, etc.) nécessaires au déploiement des machines virtuelles. Ces images servent de base pour instancier les VM via Nova selon les paramètres définis dans le template.

Heat agit comme un chef d’orchestre : il lit un template et déclenche automatiquement toutes les actions nécessaires auprès de Nova, Neutron, Cinder et Glance pour déployer une infrastructure complète.

C’est quoi la télémétrie ?

La télémétrie correspond à la collecte automatique et continue de données sur l’utilisation des ressources (CPU, RAM, stockage, réseau…).

👉 Elle permet de savoir qui consomme quoi, combien de ressources sont utilisées, et de garder un historique précis.

🧠 Exemple : combien de CPU une machine virtuelle utilise sur une journée.

C’est quoi le monitoring ?

Le monitoring consiste à surveiller en temps réel l’état du système pour détecter les problèmes ou anomalies.

👉 Il permet de savoir si les services fonctionnent correctement, si une machine est surchargée ou si une panne survient.

🧠 Exemple : alerter si le CPU dépasse 90% ou si un serveur tombe en panne.

⚡ Différence clé :

• Télémétrie = collecter et stocker les données
• Monitoring = surveiller et alerter en temps réel
🧠 Analogie :

• Télémétrie = les compteurs (eau, électricité)
• Monitoring = les alarmes et détecteurs

👉 Les deux travaillent ensemble pour comprendre et sécuriser le système.

Ce que permet Ceilometer :

  • • Collecter des métriques d’utilisation (CPU, RAM, disque, réseau, etc.)
  • • Mesurer la consommation des ressources pour la facturation (billing)
  • • Déclencher des alarmes quand un seuil est dépassé
  • • Stocker l’historique des métriques sur le long terme
  • • Fournir des données pour le monitoring et l’optimisation du cloud
  • • Suivre l’état de santé des ressources (instances, volumes, réseaux…)

Comment Ceilometer est-il construit ?

Ceilometer est composé de plusieurs agents spécialisés qui collectent les données et d’un moteur central qui les traite.

1. Ceilometer Polling Agent

Collecte périodique des métriques
Le Polling Agent interroge régulièrement les services OpenStack (Nova, Neutron, Cinder…) pour collecter des métriques d’usage (CPU, réseau, disque). Il s’exécute sur les nœuds de calcul et les contrôleurs afin de fournir une vision continue de l’activité de l’infrastructure.

2. Notification Agent

Collecte des événements en temps réel
Le Notification Agent écoute les événements envoyés via la file de messages (RabbitMQ). Il transforme ces notifications (ex : création ou démarrage d’une VM) en métriques exploitables, permettant de suivre l’activité du cloud sans polling.

3. Collector + Backend de Stockage

Centralisation et stockage des métriques
Le Collector agrège les données provenant des agents (polling et notifications) puis les envoie vers un backend de stockage spécialisé comme Gnocchi. Ce stockage permet de conserver l’historique, d’exécuter des requêtes, de générer des graphiques et d’alimenter les systèmes d’alerte.

Dans les versions récentes d’OpenStack, Ceilometer travaille souvent avec Gnocchi (pour le stockage) et Aodh (pour les alarmes).

Avec qui Ceilometer interagit-il ?

Ceilometer est le service de collecte de métriques. Il surveille l’utilisation des ressources dans OpenStack en récupérant des données auprès des autres services afin de mesurer, analyser et éventuellement facturer l’usage.

Nova

Métriques des ressources compute
Ceilometer collecte les métriques des machines virtuelles via Nova : utilisation CPU, mémoire (RAM), disque, réseau et temps d’exécution. Ces données permettent de suivre les performances, analyser l’usage et alimenter la facturation ou les alertes.

Neutron

Métriques réseau
Ceilometer collecte les données réseau via Neutron : bande passante, trafic entrant/sortant, nombre de paquets et activité des interfaces. Ces métriques permettent d’analyser les performances réseau, détecter les anomalies et optimiser la connectivité des ressources.

Cinder

Métriques de stockage bloc
Ceilometer collecte les métriques liées aux volumes via Cinder : capacité allouée, nombre de volumes, opérations d’entrée/sortie (IOPS), lectures/écritures et activité des disques. Ces données permettent de surveiller les performances, optimiser l’usage du stockage et anticiper les saturations.

Keystone

Authentification et multi-tenant
Ceilometer s’appuie sur Keystone pour authentifier les requêtes et associer chaque métrique aux bons utilisateurs et projets (tenants). Il garantit ainsi l’isolation des données et le contrôle des accès aux informations de monitoring.

Ceilometer agit comme un outil de surveillance : il collecte les données de tous les services pour analyser l’utilisation des ressources, surveiller les performances et permettre la facturation (billing).

Configuration de sudo sans mot de passe

Dans un environnement de lab/test comme celui-ci (installation OpenStack All-in-One), il est très pratique de ne plus taper son mot de passe à chaque commande sudo. Voici comment le configurer pour l’utilisateur user.

⚠️ Attention : Conséquences de cette configuration

  • L’utilisateur user pourra exécuter n’importe quelle commande en root sans jamais saisir de mot de passe.
  • Cela facilite grandement le travail (surtout avec Kolla-Ansible qui utilise beaucoup sudo).
  • Risque de sécurité : Si quelqu’un accède à votre compte utilisateur (via SSH, malware, etc.), il pourra immédiatement devenir root sans aucune protection.
  • Recommandation : Utilisez cette configuration uniquement sur une machine de laboratoire isolée ou personnelle. Pas sur un serveur en production exposé sur Internet.

Commandes à exécuter :

sudo visudo

Ajoutez cette ligne **tout en bas** du fichier (avant le #includedir) : 

user ALL=(ALL:ALL) NOPASSWD:ALL

Explication détaillée de la ligne :

  • user → Le nom de votre utilisateur (remplacez par votre vrai username si différent)
  • ALL=(ALL:ALL) → L’utilisateur peut exécuter des commandes en tant que n’importe quel utilisateur et n’importe quel groupe
  • NOPASSWD: → Pas de demande de mot de passe
  • ALL → Pour toutes les commandes

Enregistrez et quittez (Ctrl + O puis Entrée, puis Ctrl + X avec nano).

Testez ensuite avec une commande comme sudo ls /root : elle ne doit plus demander de mot de passe.

Configuration Réseau pour OpenStack

OpenStack a besoin de deux interfaces réseau distinctes pour bien fonctionner.

1

Interface de Gestion

ex : enp8s0 ou eth0
Cette interface reçoit une IP fixe. C’est par ici que vous accéderez à l’interface Horizon et aux APIs.

2

Interface Externe (Floating IPs)

ex : virt0
Cette interface ne doit avoir aucune adresse IP sur l’hôte. OpenStack la gérera entièrement.

1. Création de l’interface virtuelle virt0 

sudo modprobe dummy
sudo ip link add virt0 type dummy
sudo ip link set virt0 up

# Rendre permanent au démarrage
echo "dummy" | sudo tee /etc/modules-load.d/dummy.conf

2. Configuration du fichier Netplan

Créez ou modifiez le fichier :

sudo nano /etc/netplan/01-netcfg.yaml
network:
  version: 2
  ethernets:
    enp8s0:                    # ← Changez par le nom de votre interface réelle
      dhcp4: false
      dhcp6: false
      accept-ra: false
      addresses:
        - 192.168.1.99/24
      nameservers:
        addresses: [1.1.1.1, 8.8.8.8]
      routes:
        - to: default
          via: 192.168.1.1       # ← IP de votre box / routeur

    virt0:                     # Interface externe pour les IPs flottantes
      dhcp4: false
      dhcp6: false
      accept-ra: false
      addresses: []            # IMPORTANT : Aucune IP sur l'hôte

3. Application de la configuration 

sudo chmod 600 /etc/netplan/01-netcfg.yaml
sudo netplan apply

Vérifications importantes après netplan apply 

ip -c a

Ce que vous devez voir :

  • ✔ enp8s0 → IP fixe 192.168.1.99
  • ✔ virt0 → UP mais sans adresse IP

Test de connectivité : 

ping -c 3 1.1.1.1

Vous devez toujours avoir internet.

⚠️ Attention :

  • • L’interface virt0 doit rester sans IP sur l’hôte.
  • • Une erreur dans Netplan peut vous déconnecter en SSH.
  • • Vérifiez toujours avec ip -c a après avoir appliqué la configuration.

Rendre l’interface virt0 (Dummy) Permanente

Par défaut, l’interface virt0 créée avec ip link add disparaît après chaque redémarrage. Voici la méthode la plus stable et recommandée sur Ubuntu 24.04 pour OpenStack.

Méthode recommandée : Utilisation de systemd-networkd

1 Créer les fichiers de configuration systemd 

sudo tee /etc/systemd/network/10-virt0.netdev > /dev/null <<EOF
[NetDev]
Name=virt0
Kind=dummy
EOF

sudo tee /etc/systemd/network/20-virt0.network > /dev/null <<EOF
[Match]
Name=virt0

[Network]
# Aucune adresse IP sur l'hôte (OpenStack prendra le contrôle)
LinkLocalAddressing=no
IPv6AcceptRA=no
EOF

2 Activer et redémarrer systemd-networkd 

sudo systemctl enable systemd-networkd
sudo systemctl restart systemd-networkd

3 Vérifier que l’interface est bien permanente 

ip -c a show virt0

Résultat attendu :

virt0: <...> state UP

L’interface doit apparaître sans adresse IP.

Test final :

  1. 1. Redémarrez votre machine : sudo reboot
  2. 2. Une fois reconnecté, exécutez : ip -c a show virt0
  3. 3. Vérifiez que virt0 est toujours présente et en état UP.
Pourquoi cette méthode ?
Elle est plus fiable que les anciens fichiers /etc/modules-load.d et scripts rc.local. C’est la façon recommandée sur Ubuntu 24.04 pour créer des interfaces dummy persistantes utilisées par OpenStack (Neutron).

sudo fdisk -l

Repérez le disque ou la partition que vous voulez utiliser pour les volumes Cinder (exemple : /dev/sdb ou /dev/sdb1).

2. Installation de LVM2 

sudo apt update
sudo apt install lvm2 -y

3. Création du Physical Volume (PV) 

sudo pvcreate /dev/sdb1

Remplacez /dev/sdb1 par votre partition réelle.

4. Création du Volume Group Cinder

Attention : Nom obligatoire

Le Volume Group doit absolument s’appeler cinder-volumes.
Kolla-Ansible ne reconnaîtra pas un autre nom. 

sudo vgcreate cinder-volumes /dev/sdb1

5. Vérification finale 

sudo vgs

Résultat attendu :

VG              #PV #LV #SN Attr   VSize   VFree
  cinder-volumes    1   0   0 wz--n-  <XX.XXg> <XX.XXg>

Attention - Données effacées !

Les commandes pvcreate et vgcreate effacent toutes les données présentes sur la partition utilisée.
Assurez-vous d’utiliser un disque ou une partition vide ou dont vous avez fait une sauvegarde.

Cette configuration sera détectée automatiquement par Kolla-Ansible si vous activez Cinder dans le fichier globals.yml avec :
enable_cinder: "yes"

Configuration principale — globals.yml

Le fichier globals.yml est le cœur de la configuration de Kolla-Ansible. C’est ici que vous définissez la distribution, les interfaces réseau, l’adresse VIP et les services à activer.

sudo sed -i 's/^#\?network_interface:.*/network_interface: "enp8s0"/' /etc/kolla/globals.yml

sudo sed -i 's/^#\?enable_cinder_backend_lvm:.*/enable_cinder_backend_lvm: "yes"/' /etc/kolla/globals.yml

Paramètres essentiels à configurer

kolla_base_distro: "ubuntu"

Définit la distribution de base utilisée pour les conteneurs Docker.

kolla_internal_vip_address: "192.168.1.222"

Adresse IP virtuelle (VIP) utilisée par HAProxy pour accéder à Horizon et aux APIs OpenStack.
Choisissez une IP libre sur votre réseau local.

network_interface: "enp8s0"

Interface réseau utilisée pour le trafic de gestion (API, Horizon, base de données, etc.).

neutron_external_interface: "virt0"

Interface dédiée au réseau externe (Floating IPs). Elle ne doit avoir aucune IP sur l’hôte.

enable_cinder: "yes"

Active le service de stockage par blocs (volumes persistants).

enable_cinder_backend_lvm: "yes"

Utilise le backend LVM que nous avons configuré précédemment avec le Volume Group cinder-volumes.

Vérifier la configuration appliquée 

sudo cat /etc/kolla/globals.yml | grep -v ^# | grep -v ^$

Cette commande affiche uniquement les lignes actives (sans les commentaires ni les lignes vides).

Résumé des paramètres recommandés :

kolla_base_distro : "ubuntu"
kolla_internal_vip_address : "192.168.1.222"
network_interface : "enp8s0"
neutron_external_interface : "virt0"
enable_cinder : "yes"
enable_cinder_backend_lvm : "yes"

Nous arrivons à l’étape finale : le déploiement réel des services OpenStack via Kolla-Ansible. Cette phase peut prendre entre 15 et 45 minutes selon les performances de votre machine.

C’est la commande principale qui va télécharger les images Docker et démarrer tous les conteneurs OpenStack.

⚠️ Conseils pendant le déploiement :

  • • Ne pas interrompre le déploiement (sauf en cas d’erreur bloquante).
  • • Si le déploiement échoue, relancez les prechecks puis le deploy.

Création de la Première Machine Virtuelle (Ubuntu 24.04)

Nous allons utiliser une vraie image Ubuntu 24.04 LTS (Noble Numbat) au lieu de Cirros pour tester OpenStack dans des conditions réalistes.

1 Télécharger l’image officielle Ubuntu 24.04 Cloud 

source ~/kolla-venv/bin/activate

# Téléchargement de l'image Ubuntu 24.04 LTS (version minimale cloud)
wget https://cloud-images.ubuntu.com/releases/24.04/release/ubuntu-24.04-server-cloudimg-amd64.img

# Importer l'image dans Glance
openstack image create --disk-format qcow2 --container-format bare \
  --property os_distro=ubuntu --property os_version=24.04 \
  --public --file ubuntu-24.04-server-cloudimg-amd64.img "Ubuntu 24.04 LTS"

Lister les images 

openstack image list

2 Créer le réseau externe (Provider Network) 

# Créer le réseau externe
openstack network create --external --provider-physical-network physnet1 \
  --provider-network-type flat --share "external-net"

# Créer le subnet externe
openstack subnet create --network external-net --subnet-range 192.168.1.0/24 \
  --gateway 192.168.1.1 --allocation-pool start=192.168.1.100,end=192.168.1.200 \
  --dns-nameserver 1.1.1.1 --no-dhcp "external-subnet"

3 Créer le réseau interne et le routeur 

# Réseau interne
openstack network create internal-net
openstack subnet create --network internal-net --subnet-range 10.0.0.0/24 \
  --dns-nameserver 1.1.1.1 "internal-subnet"

# Créer et configurer le routeur
openstack router create router1
openstack router add subnet router1 internal-subnet
openstack router set --external-gateway external-net router1

4 Création du Security Group (Autoriser SSH + Ping) 

# Créer un security group dédié
openstack security group create allow-ssh-icmp --description "Autorise SSH et ICMP"

# Autoriser l'entrée SSH (Ingress) - Port 22 depuis n'importe où
openstack security group rule create --proto tcp --dst-port 22 \
  --remote-ip 0.0.0.0/0 --ingress allow-ssh-icmp

# Autoriser l'entrée ICMP (Ping) depuis n'importe où
openstack security group rule create --proto icmp \
  --remote-ip 0.0.0.0/0 --ingress allow-ssh-icmp

# (Optionnel) Autoriser la sortie vers Internet (Egress) - Généralement déjà présent par défaut
openstack security group rule create --proto tcp --dst-port 1:65535 \
  --remote-ip 0.0.0.0/0 --egress allow-ssh-icmp

Explications des règles :

  • --ingress → Trafic entrant vers la VM (important pour SSH et Ping)
  • --egress → Trafic sortant de la VM (vers Internet)
  • La plupart des security groups ont déjà une règle egress par défaut permettant tout le trafic sortant.

Lister les Security Group 

openstack security group list

Supprimer un Security Group 

openstack security group delete ID

Lister les règles dans un Security Group 

openstack security group rule list allow-ssh-icmp

Supprimer une règle de Security Group 

openstack security group rule delete ID_DE_LA_REGLE

5 Créer une paire de clés SSH 

# Créer la clé (si vous n'en avez pas déjà une)
openstack keypair create my-ssh-key > ~/.ssh/my-ssh-key.pem
chmod 600 ~/.ssh/my-ssh-key.pem

Lister les clés SSH 

openstack keypair list

Supprimer une clé SSH 

openstack keypair delete my-ssh-key

Attention :

  • Supprimer une clé SSH ne supprime pas le fichier .pem sur ton ordinateur.
  • Si tu supprimes une clé déjà utilisée par des VMs, tu ne pourras plus t’y connecter avec cette clé.
  • Il est recommandé de garder au moins une clé SSH active.

6 Création des Flavors (Modèles de VM)

Les flavors définissent la quantité de vCPU, RAM et disque pour vos machines virtuelles. 

# Flavors recommandés pour tester

# Petit (1 vCPU - 2 Go RAM)
openstack flavor create --vcpus 1 --ram 2048 --disk 20 --public m1.small

# Moyen (2 vCPU - 4 Go RAM) → Recommandé pour Ubuntu 24.04
openstack flavor create --vcpus 2 --ram 4096 --disk 30 --public m1.medium

# Grand (4 vCPU - 8 Go RAM)
openstack flavor create --vcpus 4 --ram 8192 --disk 50 --public m1.large

Lister les flavors 

openstack flavor list

Supprimer un flavor 

openstack flavor delete m1.small

7 Lancer la première instance Ubuntu 24.04 

openstack server create --flavor m1.medium --image "Ubuntu 24.04 LTS" \
  --network internal-net --key-name my-ssh-key --security-group allow-ssh-icmp \
  --min 1 --max 1 "ubuntu-24-vm"

# Suivre la création
openstack server list
openstack server show ubuntu-24-vm

8 Créer et associer une IP Flottante 

# Créer une IP flottante
openstack floating ip create external-net

# Associer l'IP flottante à la VM (remplacez par votre IP flottante)
openstack server add floating ip ubuntu-24-vm 192.168.1.XXX

# Vérifier
openstack floating ip list
openstack server show ubuntu-24-vm

# Affichage plus détaillé (recommandé)
openstack floating ip list -c ID -c "Floating IP Address" -c "Fixed IP Address" -c Port -c "Status"

Comment interpréter le résultat :

  • Fixed IP Address = vide → L’IP flottante est libre
  • Fixed IP Address = une adresse IP → L’IP flottante est déjà attribuée à une VM
  • Status = DOWN → IP flottante non utilisée
    Status = ACTIVE → IP flottante utilisée

Commandes utiles supplémentaires : 

# Supprimer une IP flottante (attention : la dissocie automatiquement)
openstack floating ip delete 192.168.1.XXX

# Dissocier une IP flottante d'une VM sans la supprimer
openstack server remove floating ip ubuntu-24-vm 192.168.1.XXX

Connexion à la VM Ubuntu 24.04 une fois l’IP flottante associée : 

ssh -i ~/.ssh/my-ssh-key.pem ubuntu@192.168.1.XXX

# Version recommandée (plus confortable)
ssh -i ~/.ssh/my-ssh-key.pem \
    -o StrictHostKeyChecking=no \
    -o UserKnownHostsFile=/dev/null \
    ubuntu@192.168.1.XXX
Astuces & Dépannage en cas de problème

Le nom d’utilisateur par défaut pour les images Ubuntu Cloud est ubuntu.

L’image Ubuntu Cloud n’a pas de mot de passe par défaut. L’authentification se fait uniquement par clé SSH.

Tu dois spécifier le chemin vers ta clé privée avec l’option -i.

La première connexion peut prendre entre 30 et 90 secondes (cloud-init est en train de configurer le système).

• "Permission denied (publickey)"

Vérifiez que le fichier de clé a les bons droits :

chmod 600 ~/.ssh/my-ssh-key.pem

• La VM reste en statut "BUILD" ou "SPAWNING" longtemps

Vérifiez les logs de la VM :

openstack console log show ubuntu-24-vm

• Impossible de se connecter même avec la bonne clé

Vérifiez que le Security Group est bien appliqué et que la règle SSH est en ingress.

• "No route to host" ou timeout

Vérifiez que l’IP flottante est bien associée et que la VM a bien démarré.

Astuce utile :

Ajoutez cette ligne dans votre ~/.ssh/config pour simplifier les connexions futures : 

Host openstack-vm
    HostName 192.168.1.XXX
    User ubuntu
    IdentityFile ~/.ssh/my-ssh-key.pem
    StrictHostKeyChecking no
