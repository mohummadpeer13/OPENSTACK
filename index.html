Infrastructure as Code

Démystifier OpenStack

Avant l’arrivée d’OpenStack, gérer une infrastructure informatique signifiait intervenir directement sur les serveurs physiques : installer les systèmes, configurer les disques, brancher les réseaux… Chaque opération demandait du temps et des manipulations manuelles.

OpenStack, c’est la couche logicielle qui prend le contrôle de tout ce matériel pour le transformer en un Cloud privé.

Face à ces contraintes, des solutions comme OpenStack ont été développées pour automatiser et simplifier la gestion des ressources. Il permet de transformer une infrastructure physique en un environnement cloud, où les machines peuvent être créées, modifiées et supprimées en quelques clics, sans intervention directe sur le matériel.

Grâce à lui, vos processeurs, votre RAM et vos disques durs deviennent des ressources virtuelles que vous pouvez piloter, découper et distribuer à vos utilisateurs en quelques secondes depuis une interface web.

De la Virtualisation au Cloud

La Virtualisation est la fondation : elle permet de découper un serveur physique en plusieurs "tranches" (VM). Mais gérer 500 serveurs à la main reste un cauchemar.

Le Cloud Computing, c'est l'étape supérieure. C'est l'automatisation totale de ces ressources. On ne parle plus de gérer des machines, mais de consommer des services via une API ou une interface, sans jamais toucher au matériel.

Le Cloud Operating System

"Comme Windows gère votre PC, OpenStack gère des datacenters entiers. RAM, CPU et Disques de milliers de serveurs fusionnent pour devenir un seul ordinateur géant."

Cloud Public

C'est le modèle de la location. Vous utilisez l'infrastructure d'un géant (AWS, Azure, GCP).

  • ⚡ Zéro maintenance matérielle
  • 💳 Coût à l'usage
  • 🌍 Présence mondiale immédiate

Cloud Privé

C'est le modèle de la propriété. Vous installez OpenStack sur vos propres machines.

  • 🛡️ Maîtrise totale des données
  • ⚙️ Personnalisation illimitée
  • 🛑 Indépendance vis-à-vis des GAFAM

Services Core

Nova

COMPUTE_ENGINE

Nova est le service principal d’OpenStack chargé de fournir des ressources de calcul (compute). On l’appelle souvent le cerveau du cloud, car il permet aux utilisateurs d’accéder en libre-service (self-service) à des machines virtuelles de manière rapide, scalable et massive.

Que permet Nova ?

  • • Créer, démarrer, arrêter, redimensionner ou supprimer des machines virtuelles en libre-service
  • • Migrer des VMs d’un serveur à un autre sans interruption
  • • Gérer des serveurs physiques (bare metal) via Ironic
  • • Supporter de nombreux hyperviseurs : KVM (le plus courant), Xen, VMware, Hyper-V, etc.

Comment Nova est-il construit ?

Nova est conçu pour être très scalable et fiable. Il repose sur une architecture basée sur des messages.

Tous les composants de Nova communiquent entre eux via une file d’attente de messages (Message Queue).

Message Queue
Généralement RabbitMQ ou Redis

Cette architecture apporte plusieurs avantages clés :

  • Scalabilité horizontale
    On peut ajouter facilement de nouveaux serveurs sans tout casser.
  • Tolérance aux pannes
    Si un composant tombe, les autres continuent de fonctionner.
  • Livraison garantie des messages
    Aucun message n’est perdu, même en cas de surcharge.
  • Traitement par priorité
    Les tâches urgentes sont traitées en priorité.

Grâce à ce système de file d’attente, Nova peut gérer des milliers de machines virtuelles sur des dizaines de serveurs physiques sans problème de performance.

Les principaux composants internes de Nova

Nova n’est pas un seul gros bloc monolithique. Il est composé de plusieurs services spécialisés qui travaillent ensemble de manière coordonnée.

Nova API

C’est le point d’entrée principal de Nova. Il reçoit toutes les demandes via HTTP (création de VM, arrêt, redimensionnement…) venant des utilisateurs, du Dashboard Horizon ou d’autres services OpenStack.

Nova Scheduler

L’ordonnanceur intelligent de Nova. Il analyse les ressources disponibles sur tous les serveurs physiques et décide sur quel nœud il est le plus judicieux de placer chaque nouvelle machine virtuelle (en fonction des règles, filtres et disponibilités).

Nova Conductor

Il joue le rôle d’intermédiaire sécurisé entre les autres composants et la base de données. Il empêche les serveurs de calcul d’accéder directement à la BDD, ce qui améliore la sécurité et les performances.

Nova Compute

Installé sur chaque nœud de calcul (serveur physique). C’est lui qui parle directement à l’hyperviseur (KVM, Xen, VMware…) pour créer, démarrer, arrêter et gérer concrètement les machines virtuelles.

Base de données SQL

Stocke l’état complet du système : toutes les instances en cours, les flavors, les métadonnées, l’historique, etc. Tous les composants passent par Nova Conductor pour y accéder.

Ces composants communiquent entre eux via une file d’attente de messages (RabbitMQ), ce qui rend Nova très scalable et résilient.

Avec qui Nova interagit-il ?

Nova ne travaille jamais seul. Il communique en permanence avec plusieurs services clés d’OpenStack pour fournir une expérience complète et cohérente.

Keystone

Authentification & Autorisations
Nova demande à Keystone de valider l’identité de l’utilisateur et de vérifier s’il a les droits nécessaires pour créer ou modifier une machine virtuelle.

Glance

Service d’Images
Nova récupère les images de machines virtuelles (Ubuntu, CentOS, Windows, etc.) stockées dans Glance pour créer de nouvelles instances.

Neutron

Réseautage & Connectivité
Nova demande à Neutron de créer des ports réseau, d’attribuer des adresses IP et de connecter les machines virtuelles aux réseaux virtuels.

Cinder

Stockage Bloc Persistant
Nova peut demander à Cinder d’attacher des volumes (disques durs virtuels persistants) aux machines virtuelles, comme un disque supplémentaire que l’on peut formater.

Placement

Suivi fin des ressources
Placement est le service moderne qui permet à Nova de connaître en temps réel les ressources disponibles (CPU, RAM, disque, GPU…) sur chaque serveur physique. Il remplace progressivement l’ancien scheduler pour une allocation plus intelligente et précise.

Ces interactions se font via des appels API et la file d’attente de messages (RabbitMQ). C’est cette collaboration entre services qui fait d’OpenStack un vrai cloud modulaire et puissant.

Neutron

NETWORKING_SERVICE

Neutron est le service Réseau d’OpenStack. On l’appelle souvent « Networking as a Service ».

Son rôle est de fournir à tous les utilisateurs un réseau virtuel complet et flexible : réseaux, sous-réseaux, adresses IP, routeurs, firewalls, load balancers, etc., en libre-service, comme sur AWS ou Google Cloud.

Ce que permet Neutron :

  • • Créer des réseaux privés (tenant networks) et des sous-réseaux
  • • Attribuer des adresses IP aux machines virtuelles
  • • Créer des routeurs virtuels et connecter plusieurs réseaux
  • • Fournir des Floating IPs pour accéder aux VMs depuis Internet
  • • Configurer des Security Groups (firewalls)
  • • Supporter des technologies avancées : VLAN, VXLAN, GRE, etc.

Comment Neutron est-il construit ?

Neutron est un service standalone et hautement scalable. Il déploie plusieurs processus sur différents nœuds (controller, network nodes et compute nodes).

Tous ces processus communiquent entre eux via une file d’attente de messages (RabbitMQ) et avec la base de données. Neutron est très modulaire grâce à son système de plugins et mechanism drivers (ML2 avec Open vSwitch, OVN, etc.).

Les principaux composants de Neutron

Neutron Server

Le cœur du service. C’est un démon Python qui expose l’API REST. Il reçoit toutes les demandes réseau et communique avec la base de données.

L2 Agent (ex: OVS Agent)

Tourne sur chaque nœud de calcul. Il configure le commutateur virtuel local (Open vSwitch, Linux Bridge…) pour connecter les machines virtuelles au réseau.

DHCP Agent

Fournit le service DHCP aux réseaux. Il attribue automatiquement des adresses IP aux machines virtuelles.

L3 Agent

Gère le routage (Layer 3) : routeurs virtuels, NAT, Floating IPs et l’accès vers les réseaux externes.

Note : Avec le driver OVN (moderne), beaucoup d’agents traditionnels sont remplacés par ovn-controller, ce qui simplifie l’architecture.

Avec qui Neutron interagit-il ?

Neutron travaille en étroite collaboration avec les autres services pour connecter les machines virtuelles au réseau.

Nova

Neutron fournit les ports réseau aux machines virtuelles créées par Nova.

Keystone

Pour l’authentification et la gestion des permissions sur les ressources réseau.

Glance

Indirectement : les images lancées par Nova utilisent les réseaux fournis par Neutron.

Cinder

Indirectement via Nova : les volumes sont attachés à des VMs connectées au réseau Neutron.

Glance

IMAGE_SERVICE

Glance est le service de gestion et de registre d’images d’OpenStack. C’est le catalogue central où sont stockées toutes les images utilisées dans le cloud.

Ce que permet Glance :

  • • Stocker et gérer les images de machines virtuelles
  • • Fournir des images prêtes à l’emploi (Ubuntu, Debian, Rocky Linux, Windows…)
  • • Gérer les snapshots des machines virtuelles
  • • Permettre aux utilisateurs de créer et d’uploader leurs propres images personnalisées
  • • Stocker les images à distance (Swift, Ceph, S3…) pour économiser l’espace disque local

Comment Glance est-il construit ?

Glance est un service relativement léger et bien structuré. Il sépare clairement la gestion des images de leur stockage physique, ce qui le rend flexible et scalable.

Glance API

C’est le **point d’entrée** de Glance. Il expose une API REST qui permet à Nova, Horizon et aux utilisateurs de lister, uploader, télécharger, supprimer ou rechercher des images. Il gère également l’authentification via Keystone.

Glance Store

C’est la partie qui s’occupe du **stockage réel** des images. Il agit comme une couche d’abstraction entre Glance et les différents systèmes de stockage.

Il supporte plusieurs backends de stockage :

• Swift (le plus recommandé)
• Ceph
• Amazon S3
• Système de fichiers local

Grâce à cette architecture modulaire, Glance peut facilement s’adapter à différents environnements de stockage tout en restant simple d’utilisation pour les autres services comme Nova.

Avec qui Glance interagit-il ?

Nova

Nova récupère les images depuis Glance pour créer et démarrer les machines virtuelles.

Keystone

Pour l’authentification et la gestion des droits d’accès aux images.

Swift / Ceph / S3

Glance délègue le stockage réel des images à un backend externe. Swift est le plus souvent utilisé.

Cinder

BLOCK_STORAGE

Cinder est le service de Stockage Bloc (Block Storage) d’OpenStack. Il permet de créer et de gérer des disques durs virtuels persistants que l’on peut attacher aux machines virtuelles.

Ce que permet Cinder :

  • • Créer des volumes (disques virtuels) de n’importe quelle taille
  • • Attacher et détacher ces volumes à des machines virtuelles (comme un disque dur externe)
  • • Créer des snapshots de volumes pour faire des sauvegardes
  • • Étendre (agrandir) un volume existant
  • • Réutiliser un volume sur une autre instance après la suppression de la précédente
  • • Utiliser différents types de stockage (SSD, HDD, Ceph, etc.)

C’est quoi le stockage bloc ?

Imagine que tu branches un disque dur externe sur ton ordinateur. Cinder fait exactement la même chose dans le cloud : il fournit des volumes persistants que tu peux attacher à une machine virtuelle.

Contrairement au stockage objet (Swift), le stockage bloc est conçu pour être utilisé comme un disque classique : tu peux le formater, installer un système de fichiers (ext4, NTFS…) et l’utiliser pour des bases de données, des applications critiques ou des données transactionnelles.

Comment Cinder est-il construit ?

Cinder est un service modulaire qui sépare clairement la gestion des volumes de leur stockage réel. Cette architecture lui permet de supporter de nombreux types de stockage en même temps (Ceph, LVM, SSD, stockage entreprise, etc.).

1. Cinder API

C’est le point d’entrée principal de Cinder. Il expose une API REST qui reçoit toutes les demandes (création de volume, attachement, détachement, suppression, etc.) provenant de Horizon, Nova ou des utilisateurs.

2. Cinder Scheduler

L’ordonnanceur intelligent de Cinder. Il analyse les capacités disponibles sur les différents backends de stockage et décide sur quel backend il est le plus judicieux de créer chaque nouveau volume.

3. Cinder Volume Service

C’est le composant qui fait le travail concret. Il communique directement avec le backend de stockage réel (Ceph, LVM, Pure Storage, NetApp, etc.) pour créer, attacher, détacher et gérer physiquement les volumes.

Grâce à cette architecture modulaire, Cinder peut travailler avec plusieurs technologies de stockage en même temps et s’adapter facilement à l’évolution de ton infrastructure.

Avec qui Cinder interagit-il ?

Nova

Nova demande à Cinder d’attacher ou de détacher un volume sur une machine virtuelle.

Keystone

Pour l’authentification et les permissions sur les volumes.

Glance

Indirectement : on peut créer un volume à partir d’une image Glance.

Swift

OBJECT_STORAGE

Swift est le service de Stockage Objet d’OpenStack. On l’appelle souvent le « Amazon S3 » open-source.

Ce que permet Swift :

  • • Stocker des quantités massives de données non structurées (images, vidéos, logs, backups, documents…)
  • • Accéder aux fichiers de n’importe où dans le monde via une API simple
  • • Garantir une très haute disponibilité et tolérance aux pannes
  • • Faire scaler le stockage presque à l’infini sans interruption
  • • Vérifier l’intégrité des données avec des checksums automatiques

C’est quoi le stockage objet ?

Contrairement au stockage bloc (Cinder) qui ressemble à un disque dur, ou au stockage fichier qui utilise des dossiers et des arborescences, Swift utilise un espace d’adressage plat (flat namespace).

Un objet (fichier) ne peut pas être « dans » un autre objet. Tous les objets existent au même niveau dans un grand pool de stockage. On les organise simplement avec des containers (comme des dossiers virtuels).

Exemple : une photo s’appelle simplement images/vacances/2025.jpg mais ce n’est pas un vrai dossier, c’est juste une clé.

Comment Swift est-il construit ?

Swift est conçu dès le départ pour être extrêmement scalable, redondant et tolérant aux pannes. Il distribue les données sur de nombreux serveurs physiques.

Proxy Server

Point d’entrée unique. Il reçoit toutes les requêtes et les route vers les bons serveurs de stockage.

Account Server

Gère les comptes et les containers (qui appartient à qui).

Container + Object Servers

Stockent réellement les données et assurent la réplication automatique.

Swift utilise un algorithme de répartition appelé Ring qui permet de savoir très rapidement sur quels serveurs se trouve chaque objet.

Avec qui Swift interagit-il ?

Glance

Swift est le backend de stockage par défaut pour les images de Glance.

Cinder

Peut utiliser Swift pour stocker des backups de volumes.

Keystone

Pour l’authentification et la gestion des permissions d’accès aux objets.

Keystone

Le Vigile : gère l’authentification, les utilisateurs, les projets, les rôles et délivre les tokens de sécurité.

IDENTITY_API

Heat

L’Orchestrateur : déploie des infrastructures complexes (stacks) en un seul clic grâce à des templates YAML.

ORCHESTRATION

Ceilometer

Le Comptable & Observateur : collecte les métriques d’utilisation pour le monitoring, l’alerting et la facturation.

TELEMETRY

Horizon

Le Tableau de bord : interface web graphique qui permet de gérer tout OpenStack (VM, réseaux, stockage…) sans ligne de commande.

DASHBOARD

Note de fin de session

OpenStack n’est pas juste un outil technique de plus.

C’est un véritable changement de paradigme dans la façon de penser et de gérer l’infrastructure informatique.

Avant OpenStack, on gérait du matériel physique : on achetait des serveurs, on les installait dans une salle, on configurait chaque machine à la main, et on demandait la permission au service informatique pour créer une nouvelle machine.

Avec OpenStack, tout change :

On ne gère plus du hardware, on pilote du logiciel qui contrôle le hardware à grande échelle.

Les développeurs peuvent créer une machine virtuelle, un réseau ou un disque en un clic, en libre-service, sans attendre des semaines. L’infrastructure devient programmable, scalable et automatisée.

En résumé :

OpenStack transforme votre infrastructure classique en un vrai Cloud Privé moderne, comparable à AWS ou Google Cloud, mais entièrement contrôlé par votre entreprise.

C’est exactement pour cette raison qu’OpenStack est l’une des meilleures portes d’entrée vers les métiers d’aujourd’hui et de demain : Architecte Cloud, Cloud Engineer et DevOps.

"Imaginez OpenStack comme le chef d'orchestre de votre centre de données. Il coordonne le calcul, le stockage et le réseau pour transformer vos serveurs physiques en une infrastructure agile et automatisée."

Pourquoi la méthode Kolla-Ansible ?

Si OpenStack est l'orchestre, Kolla-Ansible est la baguette magique qui permet de tout accorder sans fausse note. Dans ce guide, nous n'allons pas installer chaque brique manuellement. Nous allons utiliser :

  • 2
    Ansible Pour automatiser le déploiement et la configuration en une commande.

L'objectif est simple : rendre l'installation d'OpenStack non seulement accessible, mais surtout facile à maintenir.

Vérification des ressources de la machine

OpenStack Kolla-Ansible est assez gourmand en ressources, surtout en mode All-in-One. Voici les minimums fortement recommandés pour une expérience fluide :

Nombre de processeurs (CPU)

Minimum conseillé : 8 cœurs 

nproc

Cette commande affiche le nombre de cœurs logiques disponibles.

Mémoire vive (RAM)

Minimum conseillé : 16 Go 

free -mh

Affiche la mémoire totale et disponible en format lisible (Go, Mo…).

Recommandations importantes

  • • Minimum absolu : 4 cœurs + 8 Go de RAM (fonctionne mais lent et limité)
  • • Recommandé pour un bon confort : 8 cœurs + 16 Go de RAM ou plus
  • • Si vous avez moins de 16 Go, évitez de lancer trop de services en même temps (désactivez Ceilometer, Tempest…)
  • • Pour un usage intensif (plusieurs VMs), privilégiez 32 Go de RAM

Exécutez ces deux commandes avant de continuer. Si vos ressources sont insuffisantes, le déploiement risque d’être très lent ou d’échouer pendant les prechecks.

Vérification de l'accélération matérielle KVM

OpenStack utilise KVM comme hyperviseur par défaut. Il est essentiel que l’accélération matérielle soit activée pour avoir de bonnes performances.

1. Installation et vérification KVM 

sudo apt install cpu-checker -y
kvm-ok

✅ Résultat attendu :

KVM acceleration can be used

2. Chargement du module noyau KVM 

sudo modprobe kvm-intel

(Utilisez kvm-amd si vous avez un processeur AMD)

3. Vérification de la virtualisation imbriquée (Nested Virtualization) 

cat /sys/module/kvm_intel/parameters/nested

✅ Résultat attendu :

Y

La virtualisation imbriquée est activée (recommandé pour les labs OpenStack).

Si vous obtenez "N" :

Vous pouvez l’activer temporairement avec :
sudo modprobe -r kvm_intel && sudo modprobe kvm_intel nested=1

Pourquoi ces vérifications sont importantes ?

  • • Sans accélération KVM, vos machines virtuelles seront très lentes (émulation logicielle).
  • • La virtualisation imbriquée (nested) permet de lancer des VMs dans des VMs (utile pour tester Kubernetes, OpenShift, etc. dans OpenStack).
  • • Kolla-Ansible détecte automatiquement KVM, mais mieux vaut vérifier avant le déploiement.

Configuration de sudo sans mot de passe

Dans un environnement de lab/test comme celui-ci (installation OpenStack All-in-One), il est très pratique de ne plus taper son mot de passe à chaque commande sudo. Voici comment le configurer pour l’utilisateur user.

⚠️ Attention : Conséquences de cette configuration

  • L’utilisateur user pourra exécuter n’importe quelle commande en root sans jamais saisir de mot de passe.
  • Cela facilite grandement le travail (surtout avec Kolla-Ansible qui utilise beaucoup sudo).
  • Risque de sécurité : Si quelqu’un accède à votre compte utilisateur (via SSH, malware, etc.), il pourra immédiatement devenir root sans aucune protection.
  • Recommandation : Utilisez cette configuration uniquement sur une machine de laboratoire isolée ou personnelle. Pas sur un serveur en production exposé sur Internet.

Commandes à exécuter :

sudo visudo

Ajoutez cette ligne **tout en bas** du fichier (avant le #includedir) : 

user ALL=(ALL:ALL) NOPASSWD:ALL

Explication détaillée de la ligne :

  • user → Le nom de votre utilisateur (remplacez par votre vrai username si différent)
  • ALL=(ALL:ALL) → L’utilisateur peut exécuter des commandes en tant que n’importe quel utilisateur et n’importe quel groupe
  • NOPASSWD: → Pas de demande de mot de passe
  • ALL → Pour toutes les commandes

Enregistrez et quittez (Ctrl + O puis Entrée, puis Ctrl + X avec nano).

Testez ensuite avec une commande comme sudo ls /root : elle ne doit plus demander de mot de passe.

Configuration Réseau pour OpenStack

OpenStack a besoin de deux interfaces réseau distinctes pour bien fonctionner.

1

Interface de Gestion

ex : enp8s0 ou eth0
Cette interface reçoit une IP fixe. C’est par ici que vous accéderez à l’interface Horizon et aux APIs.

2

Interface Externe (Floating IPs)

ex : virt0
Cette interface ne doit avoir aucune adresse IP sur l’hôte. OpenStack la gérera entièrement.

1. Création de l’interface virtuelle virt0 

sudo modprobe dummy
sudo ip link add virt0 type dummy
sudo ip link set virt0 up

# Rendre permanent au démarrage
echo "dummy" | sudo tee /etc/modules-load.d/dummy.conf

2. Configuration du fichier Netplan

Créez ou modifiez le fichier :

sudo nano /etc/netplan/01-netcfg.yaml
network:
  version: 2
  ethernets:
    enp8s0:                    # ← Changez par le nom de votre interface réelle
      dhcp4: false
      dhcp6: false
      accept-ra: false
      addresses:
        - 192.168.1.99/24
      nameservers:
        addresses: [1.1.1.1, 8.8.8.8]
      routes:
        - to: default
          via: 192.168.1.1       # ← IP de votre box / routeur

    virt0:                     # Interface externe pour les IPs flottantes
      dhcp4: false
      dhcp6: false
      accept-ra: false
      addresses: []            # IMPORTANT : Aucune IP sur l'hôte

3. Application de la configuration 

sudo chmod 600 /etc/netplan/01-netcfg.yaml
sudo netplan apply

Vérifications importantes après netplan apply 

ip -c a

Ce que vous devez voir :

  • ✔ enp8s0 → IP fixe 192.168.1.99
  • ✔ virt0 → UP mais sans adresse IP

Test de connectivité : 

ping -c 3 1.1.1.1

Vous devez toujours avoir internet.

⚠️ Attention :

  • • L’interface virt0 doit rester sans IP sur l’hôte.
  • • Une erreur dans Netplan peut vous déconnecter en SSH.
  • • Vérifiez toujours avec ip -c a après avoir appliqué la configuration.

Rendre l’interface virt0 (Dummy) Permanente

Par défaut, l’interface virt0 créée avec ip link add disparaît après chaque redémarrage. Voici la méthode la plus stable et recommandée sur Ubuntu 24.04 pour OpenStack.

Méthode recommandée : Utilisation de systemd-networkd

1 Créer les fichiers de configuration systemd 

sudo tee /etc/systemd/network/10-virt0.netdev > /dev/null <<EOF
[NetDev]
Name=virt0
Kind=dummy
EOF

sudo tee /etc/systemd/network/20-virt0.network > /dev/null <<EOF
[Match]
Name=virt0

[Network]
# Aucune adresse IP sur l'hôte (OpenStack prendra le contrôle)
LinkLocalAddressing=no
IPv6AcceptRA=no
EOF

2 Activer et redémarrer systemd-networkd 

sudo systemctl enable systemd-networkd
sudo systemctl restart systemd-networkd

3 Vérifier que l’interface est bien permanente 

ip -c a show virt0

Résultat attendu :

virt0: <...> state UP

L’interface doit apparaître sans adresse IP.

Test final :

  1. 1. Redémarrez votre machine : sudo reboot
  2. 2. Une fois reconnecté, exécutez : ip -c a show virt0
  3. 3. Vérifiez que virt0 est toujours présente et en état UP.
Pourquoi cette méthode ?
Elle est plus fiable que les anciens fichiers /etc/modules-load.d et scripts rc.local. C’est la façon recommandée sur Ubuntu 24.04 pour créer des interfaces dummy persistantes utilisées par OpenStack (Neutron).

Configuration du Disque Dur Persistant (Cinder)

Pour permettre aux instances OpenStack de créer des volumes persistants (disques durs qui survivent même si la VM est supprimée), nous allons configurer un Volume Group LVM dédié avec Cinder.

Recommandation

Idéalement, utilisez un second disque dur (ex: /dev/sdb) ou une partition dédiée. Ce disque ne doit pas contenir de données importantes, car il sera entièrement utilisé par Cinder.

1. Vérifier les disques disponibles 

sudo fdisk -l

Repérez le disque ou la partition que vous voulez utiliser pour les volumes Cinder (exemple : /dev/sdb ou /dev/sdb1).

2. Installation de LVM2 

sudo apt update
sudo apt install lvm2 -y

3. Création du Physical Volume (PV) 

sudo pvcreate /dev/sdb1

Remplacez /dev/sdb1 par votre partition réelle.

4. Création du Volume Group Cinder

Attention : Nom obligatoire

Le Volume Group doit absolument s’appeler cinder-volumes.
Kolla-Ansible ne reconnaîtra pas un autre nom. 

sudo vgcreate cinder-volumes /dev/sdb1

5. Vérification finale 

sudo vgs

Résultat attendu :

VG              #PV #LV #SN Attr   VSize   VFree
  cinder-volumes    1   0   0 wz--n-  <XX.XXg> <XX.XXg>

Attention - Données effacées !

Les commandes pvcreate et vgcreate effacent toutes les données présentes sur la partition utilisée.
Assurez-vous d’utiliser un disque ou une partition vide ou dont vous avez fait une sauvegarde.

Cette configuration sera détectée automatiquement par Kolla-Ansible si vous activez Cinder dans le fichier globals.yml avec :
enable_cinder: "yes"

Configuration principale — globals.yml

Le fichier globals.yml est le cœur de la configuration de Kolla-Ansible. C’est ici que vous définissez la distribution, les interfaces réseau, l’adresse VIP et les services à activer.

Méthode recommandée : Modification avec sed (rapide et précise) 

# Application des paramètres essentiels avec sed
sudo sed -i 's/^#\?kolla_base_distro:.*/kolla_base_distro: "ubuntu"/' /etc/kolla/globals.yml
sudo sed -i 's/^#\?kolla_internal_vip_address:.*/kolla_internal_vip_address: "192.168.1.222"/' /etc/kolla/globals.yml
sudo sed -i 's/^#\?network_interface:.*/network_interface: "enp8s0"/' /etc/kolla/globals.yml
sudo sed -i 's/^#\?neutron_external_interface:.*/neutron_external_interface: "virt0"/' /etc/kolla/globals.yml
sudo sed -i 's/^#\?enable_cinder:.*/enable_cinder: "yes"/' /etc/kolla/globals.yml
sudo sed -i 's/^#\?enable_cinder_backend_lvm:.*/enable_cinder_backend_lvm: "yes"/' /etc/kolla/globals.yml

Paramètres essentiels à configurer

kolla_base_distro: "ubuntu"

Définit la distribution de base utilisée pour les conteneurs Docker.

kolla_internal_vip_address: "192.168.1.222"

Adresse IP virtuelle (VIP) utilisée par HAProxy pour accéder à Horizon et aux APIs OpenStack.
Choisissez une IP libre sur votre réseau local.

network_interface: "enp8s0"

Interface réseau utilisée pour le trafic de gestion (API, Horizon, base de données, etc.).

neutron_external_interface: "virt0"

Interface dédiée au réseau externe (Floating IPs). Elle ne doit avoir aucune IP sur l’hôte.

enable_cinder: "yes"

Active le service de stockage par blocs (volumes persistants).

enable_cinder_backend_lvm: "yes"

Utilise le backend LVM que nous avons configuré précédemment avec le Volume Group cinder-volumes.

Vérifier la configuration appliquée 

sudo cat /etc/kolla/globals.yml | grep -v ^# | grep -v ^$

Cette commande affiche uniquement les lignes actives (sans les commentaires ni les lignes vides).

Résumé des paramètres recommandés :

kolla_base_distro : "ubuntu"
kolla_internal_vip_address : "192.168.1.222"
network_interface : "enp8s0"
neutron_external_interface : "virt0"
enable_cinder : "yes"
enable_cinder_backend_lvm : "yes"
## enable_haproxy : "no"

Création de la Première Machine Virtuelle (Ubuntu 24.04)

Nous allons utiliser une vraie image Ubuntu 24.04 LTS (Noble Numbat) au lieu de Cirros pour tester OpenStack dans des conditions réalistes.

1 Télécharger l’image officielle Ubuntu 24.04 Cloud 

source ~/kolla-venv/bin/activate

# Téléchargement de l'image Ubuntu 24.04 LTS (version minimale cloud)
wget https://cloud-images.ubuntu.com/releases/24.04/release/ubuntu-24.04-server-cloudimg-amd64.img

# Importer l'image dans Glance
openstack image create --disk-format qcow2 --container-format bare \
  --property os_distro=ubuntu --property os_version=24.04 \
  --public --file ubuntu-24.04-server-cloudimg-amd64.img "Ubuntu 24.04 LTS"

Lister les images 

openstack image list

2 Créer le réseau externe (Provider Network) 

# Créer le réseau externe
openstack network create --external --provider-physical-network physnet1 \
  --provider-network-type flat --share "external-net"

# Créer le subnet externe
openstack subnet create --network external-net --subnet-range 192.168.1.0/24 \
  --gateway 192.168.1.1 --allocation-pool start=192.168.1.100,end=192.168.1.200 \
  --dns-nameserver 1.1.1.1 --no-dhcp "external-subnet"

3 Créer le réseau interne et le routeur 

# Réseau interne
openstack network create internal-net
openstack subnet create --network internal-net --subnet-range 10.0.0.0/24 \
  --dns-nameserver 1.1.1.1 "internal-subnet"

# Créer et configurer le routeur
openstack router create router1
openstack router add subnet router1 internal-subnet
openstack router set --external-gateway external-net router1

4 Création du Security Group (Autoriser SSH + Ping) 

# Créer un security group dédié
openstack security group create allow-ssh-icmp --description "Autorise SSH et ICMP"

# Autoriser l'entrée SSH (Ingress) - Port 22 depuis n'importe où
openstack security group rule create --proto tcp --dst-port 22 \
  --remote-ip 0.0.0.0/0 --ingress allow-ssh-icmp

# Autoriser l'entrée ICMP (Ping) depuis n'importe où
openstack security group rule create --proto icmp \
  --remote-ip 0.0.0.0/0 --ingress allow-ssh-icmp

# (Optionnel) Autoriser la sortie vers Internet (Egress) - Généralement déjà présent par défaut
openstack security group rule create --proto tcp --dst-port 1:65535 \
  --remote-ip 0.0.0.0/0 --egress allow-ssh-icmp

Explications des règles :

  • --ingress → Trafic entrant vers la VM (important pour SSH et Ping)
  • --egress → Trafic sortant de la VM (vers Internet)
  • La plupart des security groups ont déjà une règle egress par défaut permettant tout le trafic sortant.

Lister les Security Group 

openstack security group list

Supprimer un Security Group 

openstack security group delete ID

Lister les règles dans un Security Group 

openstack security group rule list allow-ssh-icmp

Supprimer une règle de Security Group 

openstack security group rule delete ID_DE_LA_REGLE

5 Créer une paire de clés SSH 

# Créer la clé (si vous n'en avez pas déjà une)
openstack keypair create my-ssh-key > ~/.ssh/my-ssh-key.pem
chmod 600 ~/.ssh/my-ssh-key.pem

Lister les clés SSH 

openstack keypair list

Supprimer une clé SSH 

openstack keypair delete my-ssh-key

Attention :

  • Supprimer une clé SSH ne supprime pas le fichier .pem sur ton ordinateur.
  • Si tu supprimes une clé déjà utilisée par des VMs, tu ne pourras plus t’y connecter avec cette clé.
  • Il est recommandé de garder au moins une clé SSH active.

6 Création des Flavors (Modèles de VM)

Les flavors définissent la quantité de vCPU, RAM et disque pour vos machines virtuelles. 

# Flavors recommandés pour tester

# Petit (1 vCPU - 2 Go RAM)
openstack flavor create --vcpus 1 --ram 2048 --disk 20 --public m1.small

# Moyen (2 vCPU - 4 Go RAM) → Recommandé pour Ubuntu 24.04
openstack flavor create --vcpus 2 --ram 4096 --disk 30 --public m1.medium

# Grand (4 vCPU - 8 Go RAM)
openstack flavor create --vcpus 4 --ram 8192 --disk 50 --public m1.large

Lister les flavors 

openstack flavor list

Supprimer un flavor 

openstack flavor delete m1.small

7 Lancer la première instance Ubuntu 24.04 

openstack server create --flavor m1.medium --image "Ubuntu 24.04 LTS" \
  --network internal-net --key-name my-ssh-key --security-group allow-ssh-icmp \
  --min 1 --max 1 "ubuntu-24-vm"

# Suivre la création
openstack server list
openstack server show ubuntu-24-vm

8 Créer et associer une IP Flottante 

# Créer une IP flottante
openstack floating ip create external-net

# Associer l'IP flottante à la VM (remplacez par votre IP flottante)
openstack server add floating ip ubuntu-24-vm 192.168.1.XXX

# Vérifier
openstack floating ip list
openstack server show ubuntu-24-vm

# Affichage plus détaillé (recommandé)
openstack floating ip list -c ID -c "Floating IP Address" -c "Fixed IP Address" -c Port -c "Status"

Comment interpréter le résultat :

  • Fixed IP Address = vide → L’IP flottante est libre
  • Fixed IP Address = une adresse IP → L’IP flottante est déjà attribuée à une VM
  • Status = DOWN → IP flottante non utilisée
    Status = ACTIVE → IP flottante utilisée

Commandes utiles supplémentaires : 

# Supprimer une IP flottante (attention : la dissocie automatiquement)
openstack floating ip delete 192.168.1.XXX

# Dissocier une IP flottante d'une VM sans la supprimer
openstack server remove floating ip ubuntu-24-vm 192.168.1.XXX

Connexion à la VM Ubuntu 24.04 une fois l’IP flottante associée : 

ssh -i ~/.ssh/my-ssh-key.pem ubuntu@192.168.1.XXX

# Version recommandée (plus confortable)
ssh -i ~/.ssh/my-ssh-key.pem \
    -o StrictHostKeyChecking=no \
    -o UserKnownHostsFile=/dev/null \
    ubuntu@192.168.1.XXX
Astuces & Dépannage en cas de problème

Le nom d’utilisateur par défaut pour les images Ubuntu Cloud est ubuntu.

L’image Ubuntu Cloud n’a pas de mot de passe par défaut. L’authentification se fait uniquement par clé SSH.

Tu dois spécifier le chemin vers ta clé privée avec l’option -i.

La première connexion peut prendre entre 30 et 90 secondes (cloud-init est en train de configurer le système).

• "Permission denied (publickey)"

Vérifiez que le fichier de clé a les bons droits :

chmod 600 ~/.ssh/my-ssh-key.pem

• La VM reste en statut "BUILD" ou "SPAWNING" longtemps

Vérifiez les logs de la VM :

openstack console log show ubuntu-24-vm

• Impossible de se connecter même avec la bonne clé

Vérifiez que le Security Group est bien appliqué et que la règle SSH est en ingress.

• "No route to host" ou timeout

Vérifiez que l’IP flottante est bien associée et que la VM a bien démarré.

Astuce utile :

Ajoutez cette ligne dans votre ~/.ssh/config pour simplifier les connexions futures : 

Host openstack-vm
    HostName 192.168.1.XXX
    User ubuntu
    IdentityFile ~/.ssh/my-ssh-key.pem
    StrictHostKeyChecking no
